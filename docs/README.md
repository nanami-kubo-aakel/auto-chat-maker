## 🎯 各ディレクトリの役割と特徴

### 1. 要件定義 (requirements/) - 「何を作るか」

**目的**: プロジェクトの目的、機能、制約を明確に定義し、ステークホルダー間で合意を形成する

**特徴**:
- **ビジネス視点**: ビジネス目標と価値提案を中心とした要件
- **機能中心**: システムが提供すべき機能を詳細に定義
- **制約明確化**: 技術的・運用上の制約を明文化
- **合意形成**: 開発チーム、ステークホルダー間での合意ベース

**含まれる資料**:
- ビジネス要件（プロジェクト背景、目的、成功指標）
- 機能要件（システム機能の詳細定義）
- 非機能要件（性能、セキュリティ、可用性）
- 制約事項（技術的・運用上の制約）
- ユースケース（システム利用シナリオ）
- 開発計画（実装スケジュール、リスク管理）

### 2. 設計書 (design/) - 「どのように作るか」

**目的**: 要件定義に基づいて、システムの具体的な実装方法を設計し、開発チームが一貫した実装を行えるようにする

**特徴**:
- **実装前の設計**: 実際の実装前にシステム全体の設計を行う
- **技術選定**: 要件を満たすための技術スタックを選定
- **アーキテクチャ設計**: システム全体の構造とコンポーネント間関係を定義
- **拡張性考慮**: 将来的な機能拡張を考慮した設計

**含まれる資料**:
- システム全体構造設計書（アーキテクチャ、データフロー）
- 技術設計書（技術スタック選定、技術構成）
- データベース設計書（テーブル定義、リレーション）
- ER図（エンティティ関係図）

### 3. 仕様書 (specification/) - 「具体的な実装方法」

**目的**: 設計書に基づいて、開発者が実際にコーディングを行う際に参照する詳細な技術仕様を定義する

**特徴**:
- **実装反映**: 実際に実装した内容が反映される
- **詳細仕様**: 実装者が迷わないレベルの詳細な仕様
- **技術詳細**: API仕様、データ形式、エラーハンドリング等の詳細
- **保守性**: 実装後の保守・運用に必要な情報

**含まれる資料**:
- 技術仕様書（API仕様、データ形式、エラーコード）
- アーキテクチャ設計書（実装詳細、ディレクトリ構造）
- デプロイメント仕様（環境設定、CI/CD）

## 🔄 開発プロセスとの関係

### 要件定義フェーズ
```
ビジネス要件 → 機能要件 → 非機能要件 → 制約事項 → ユースケース
```
- **成果物**: requirements/ ディレクトリの資料
- **参加者**: プロダクトオーナー、ビジネスアナリスト、開発チーム
- **目的**: システムの範囲と機能を明確化

### 設計フェーズ
```
要件定義 → アーキテクチャ設計 → 技術設計 → データベース設計
```
- **成果物**: design/ ディレクトリの資料
- **参加者**: アーキテクト、開発チーム
- **目的**: 実装方法の設計と技術選定

### 実装フェーズ
```
設計書 → 実装 → 仕様書更新 → テスト
```
- **成果物**: specification/ ディレクトリの資料
- **参加者**: 開発チーム
- **目的**: 実際の実装と詳細仕様の確定

## 📋 ドキュメント作成ガイドライン

### ファイル形式
- **Markdown (.md)**: 基本的なドキュメント
- **PlantUML (.puml)**: 図表（UML、ER図等）
- **YAML (.yaml)**: 設定ファイル
- **JSON (.json)**: データ定義

### 命名規則
- ファイル名は小文字とハイフンを使用（例: `business-requirements.md`）
- 日本語ファイル名は避ける
- バージョン番号は含めない（Gitで管理）

### 更新履歴
各ドキュメントには以下の情報を含める
- 初版作成日
- 最終更新日
- 更新者名
- 更新内容の要約

## 📚 ドキュメント活用方法

### 新規開発者向け
1. **requirements/README.md** - プロジェクト概要の理解
2. **design/README.md** - システム設計の理解
3. **specification/README.md** - 実装詳細の理解

### 開発チーム向け
- **要件変更時**: requirements/ の該当資料を更新
- **設計変更時**: design/ の該当資料を更新
- **実装完了時**: specification/ の該当資料を更新

### ステークホルダー向け
- **requirements/** - プロジェクトの目的と機能の理解
- **design/system-architecture.md** - システム全体の構造理解

## 🔍 品質管理

### レビュープロセス
1. **要件レビュー**: ビジネス要件の妥当性確認
2. **設計レビュー**: 技術設計の妥当性確認
3. **実装レビュー**: 実装仕様の妥当性確認

### トレーサビリティ
- 要件 → 設計 → 実装の追跡可能性を確保
- 各段階での変更影響範囲の明確化
